# 验收检查 — AC 逐条验证

## 触发条件
- 功能实现完成后
- 用户要求"验收"
- PR review 前的最终检查

## 流程

### 1. 加载规格
读取 `specs/{feature}/PRD.md` 的验收标准部分。

### 2. 逐条验证

对每个 AC (Acceptance Criteria):

```
AC-{N}: Given [前置] When [操作] Then [预期]

验证方式 (按优先级选择):
1. 自动测试 — 有对应测试用例，运行通过 ✅
2. API 测试 — curl/httpie 调用 API 验证响应 ✅
3. 代码审查 — 阅读代码确认逻辑正确 ✅
4. 手动确认 — 需要用户手动验证的标注 ⏳
```

### 3. 产出验收报告

```markdown
# 验收报告: {feature}

## 日期: YYYY-MM-DD
## 规格: specs/{feature}/PRD.md

## 验收标准检查

| AC | 描述 | 状态 | 验证方式 | 证据 |
|----|------|------|----------|------|
| AC-1 | 用户可以... | ✅ 通过 | 测试 test_xxx | 测试输出 |
| AC-2 | 当...时... | ✅ 通过 | API curl | 响应 200 |
| AC-3 | 错误处理... | ❌ 未通过 | 代码审查 | 缺少 xxx |
| AC-4 | 界面显示... | ⏳ 待手动 | 需用户确认 | — |

## 非功能检查
- [ ] 性能: 响应时间 < Xms
- [ ] 安全: 输入校验, 权限检查
- [ ] 兼容性: 浏览器/设备

## 质量检查
- [ ] 编译通过，零 warning
- [ ] 全量测试通过
- [ ] 无 console.log / debug 代码
- [ ] 无硬编码 secrets
- [ ] 文档更新

## 结论
- ✅ **验收通过** — 所有 AC 满足
- ⚠️ **有条件通过** — [N] 个 AC 需要手动确认
- ❌ **验收未通过** — [N] 个 AC 失败，需修复:
  - AC-X: [问题描述和修复建议]
```

### 4. 处理失败项
- ❌ 失败 → 回到实现，修复后重新验收
- ⏳ 待手动 → 标注清楚，交给用户确认
- ✅ 全通过 → 准备提交/合并

## 关键原则
- **AC 是合同** — 不能模糊通过，每条必须有明确的证据
- **自动优先** — 能自动验证的不要手动
- **证据留存** — 验收报告保存在 specs/ 目录
- **失败快反馈** — 发现问题立即报告，不要全部做完再说
